## 1. Problem Statement

联想渠道发票报备的核心场景是：L2 提货服务商向 L1 经销商销售后，需要通过发票向联想总部报备流向。当前流程完全依赖 L2 人工填报 + 后台人工审核，三类用户都面临显著痛点：

**L2 提货服务商的痛点：**
- 每次报备需要手动填写 20+ 字段（发票号、购买方 / 销售方、商品明细、金额、税额等），单次填报耗时 15-20 分钟；
- 缺少提交前的必填校验，常因缺字段被后台驳回，反复修改；
- 上传错了发票无法重传，只能新建一份报备单。

**后台审核员的痛点：**
- 每天需要审核数百张报备单，逐字段与发票原件比对，效率低；
- 难以快速识别"OCR 识别与 L2 提交不一致"的可疑单；
- 看不到完整的字段修改历史，无法判断 L2 是否故意篡改关键字段。

**后台主管的痛点：**
- 跨区域查看和监督下属审核员的工作成本高；
- 审核员做出错误决议后，缺少介入和纠正机制；
- 难以获取全局的审批质量数据。

这些问题导致 L2 报备体验差、后台审核成本高、异常单难以及时识别、审批过程难以审计追溯。

## 2. Solution

针对上一节列出的三方痛点，本期 MVP 通过引入 **AI OCR + 规则引擎 + 三角色审批流** 来重塑发票报备流程。核心方案是：把"AI"嵌入到 L2 填报、提交校验、后台审批三个关键节点，**让 AI 成为审核助手而非审批者，最终通过 / 驳回始终由后台人工决定**。

### 针对 L2 提货服务商

- **OCR 预填**：上传发票后，系统自动识别并预填 20+ 字段（含发票号、购销方名称 / 统一代码、商品行、金额、税额等），把 L2 录入负担降到最低；
- **提交前阻断校验**：系统自动检查必填字段，缺什么字段一目了然，避免提交后才发现缺字段被驳回；
- **发票重传**：上传错了发票可以重传，系统重新触发 OCR 后**用新识别结果覆盖之前的 OCR 内容**；L2 已编辑的字段保留不丢；
- **提交后反馈**：提交后能看到 AI 给出的"正常 / 异常"标签、异常原因、修改日志、后台审批进度，及时了解自己的报备状态。

### 针对后台审核员

- **AI 异常标签优先处理**：系统根据规则自动给每张报备单打"正常 / 异常"标签，审核员可优先处理异常单；
- **OCR 原值 vs L2 提交值对比**：报备单详情页高亮显示 OCR 识别原值与 L2 提交值的差异字段，审核员一眼能看出可疑点；
- **完整修改日志**：所有字段修改、人工补录、AI 异常判断都进入修改日志，辅助审核员判断 L2 是否故意篡改关键字段。

### 针对后台主管

- **跨区域全局可见**：主管可查看任意区域的报备单和审核员工作，不受区域限制；
- **全员审批历史**：主管可查看所有审核员处理过的单子，包括已通过、已驳回、待审批；
- **撤回任意决议**：主管可主动撤回任意审核员的"通过 / 驳回"决议，单子回到"待 L2 确认"，L2 可重新编辑提交；
- **修改任意字段**：主管在撤回决议后，单据回到"待 L2 确认"状态；若主管直接修改字段，必须记录为后台主管修改日志，并自动重新执行提交前校验与 AI 打标。

### 三个核心设计原则

1. **AI 是助手，不是审批者**：所有 AI 打标都只作为辅助参考，最终通过 / 驳回由后台人工决定；
2. **数据可追溯**：所有字段修改、AI 打标、审批决议、撤回操作都进入修改日志，事后可还原完整操作链；
3. **审批通过即锁定**：进入"已锁定 / 审核归档"状态后默认不可修改，唯一例外是主管主动撤回（撤回后回到"待 L2 确认"解锁）。

## 3. User Stories

按角色分组，覆盖 L2 提货服务商、后台审核员、后台主管三个角色从进入到结束的完整流程。

### L2 提货服务商

1. 作为 L2 提货服务商，我希望能上传发票图片或 PDF，以便系统 OCR 识别并预填报备字段。
2. 作为 L2 提货服务商，我希望 OCR 自动预填主表（23 个字段）和明细表（N 行商品），以便我不需要手动填写所有字段。
3. 作为 L2 提货服务商，我希望能编辑任何 OCR 预填字段，以便修正 OCR 错误。
4. 作为 L2 提货服务商，我希望系统在提交前自动校验必填字段，以便我不会因缺字段被后台驳回。
5. 作为 L2 提货服务商，我希望系统明确提示哪些必填字段缺失，以便我知道补什么。
6. 作为 L2 提货服务商，我希望能重传发票并重新触发 OCR，以便改正上传错的发票。
7. 作为 L2 提货服务商，我希望重传发票后用新识别结果覆盖之前的 OCR 字段，其他内容不变，以便改正上传错的发票且不影响其他字段。
8. 作为 L2 提货服务商，我希望能提交报备单，以便进入后台审批队列。
9. 作为 L2 提货服务商，我希望提交后能看到 AI 给出的"正常 / 异常"标签，以便了解系统的初步判断。
10. 作为 L2 提货服务商，我希望能看到异常原因（如"关键字段被修改"、"发票疑似重复"），以便理解为什么被打异常。
11. 作为 L2 提货服务商，我希望能看到后台审批进度（待审批 / 已通过 / 已驳回），以便知道当前状态。
12. 作为 L2 提货服务商，我希望能查看自己的修改日志，以便验证我提交的内容。
13. 作为 L2 提货服务商，我希望能仅在"待人工审批"状态撤回报备单，以便在后台处理前主动放弃。
14. 作为 L2 提货服务商，我希望撤回后能新建报备单重新提交，以便修正问题。
15. 作为 L2 提货服务商，我希望在"驳回"后能编辑并重新提交，以便修正问题后再次提交。

### 后台审核员

16. 作为后台审核员，我希望能查看所有提交的报备单，以便审核。
17. 作为后台审核员，我希望能按省份 / 地市 / 城市筛选，以便聚焦我负责的区域。
18. 作为后台审核员，我希望能区分正常单和异常单，以便优先处理异常单。
19. 作为后台审核员，我希望能查看单子详情（发票原件、OCR 原值、L2 提交值、产品明细、AI 标签、异常原因、修改日志），以便综合判断。
20. 作为后台审核员，我希望能直接看到 OCR 原值 vs L2 提交值的对比，以便一眼识别可疑修改。
21. 作为后台审核员，我希望能查看完整的修改日志，以便判断 L2 是否故意篡改关键字段。
22. 作为后台审核员，我希望能通过报备单，以便进入"已锁定 / 审核归档"状态。
23. 作为后台审核员，我希望能驳回报备单并填写原因，以便 L2 知道怎么修改。
24. 作为后台审核员，我希望能撤回自己的审批决议，以便改正错误。
25. 作为后台审核员，我希望能查看自己的处理历史，以便跟踪我的工作。

### 后台主管 / 组长

26. 作为后台主管，我希望能跨区域查看所有报备单，以便不受区域限制。
27. 作为后台主管，我希望能查看所有审核员的处理历史，以便监督工作质量。
28. 作为后台主管，我希望能查看全员的操作日志，以便审计追溯。
29. 作为后台主管，我希望能撤回任意审核员的决议，以便纠正错误。
30. 作为后台主管，我希望能撤回决议后修改任意字段，修改后自动重新触发校验与 AI 打标，并记录为后台主管修改日志，以便直接修正问题且留痕可追溯。
31. 作为后台主管，我希望保持单级审批机制（不强制介入），以便不增加流程负担；如需监督，主动查看或撤回即可。

## 4. Implementation Decisions

### 整体架构

本期 MVP 采用 **前后端分离 + 外部 OCR 服务** 的三层结构：

```
┌─────────────────────────────────────────────────┐
│                 Frontend (Web)                   │
│  L2 端 (移动端 H5)          后台 (PC Web)         │
│  Vue 3 + Vant              Vue 3 + Element Plus   │
└─────────────────────────────────────────────────┘
                       ↓ HTTPS + JSON
┌─────────────────────────────────────────────────┐
│              Backend API Service                 │
│              Node.js + NestJS                    │
│  (Auth / Report / OCR / Validation / AI-Label)   │
└─────────────────────────────────────────────────┘
            ↓                  ↓                ↓
┌──────────────┐  ┌──────────────────┐  ┌──────────────┐
│   MySQL 8.0  │  │      MinIO       │  │  百度 OCR API │
│  (业务数据)    │  │  (发票文件存储)     │  │  (外部 OCR)   │
└──────────────┘  └──────────────────┘  └──────────────┘
                       ↑
              联想现有 SSO (OIDC)
```

### 关键接口契约（API Contract）

所有接口走 HTTPS + JSON，鉴权采用 JWT Token。

#### 4.4.1 上传发票 + 触发 OCR

```
POST /api/reports/{id}/invoice
请求：multipart/form-data，字段 file
响应：
  {
    "report_id": "...",
    "ocr_status": "识别成功 / 识别失败 / 部分识别",
    "ocr_fields": { ... },
    "ocr_confidence": 0.95
  }
```

#### 4.4.2 重传发票（覆盖 OCR）

```
POST /api/reports/{id}/invoice/reupload
请求：multipart/form-data，字段 file
行为：
  - 清空所有 ocr_* 字段
  - 重新触发 OCR
  - L2 已编辑的字段保留
  - 状态保持"待 L2 确认"
响应：同 4.4.1
```

#### 4.4.3 提交报备单（含必填校验）

```
POST /api/reports/{id}/submit
请求：{
  "fields": { ... }  // L2 提交的所有字段
}
响应：
  - 200: 成功，进入"待人工审批-正常/异常"
  - 400: 必填字段缺失，返回缺失字段列表
```

#### 4.4.4 后台审批

```
POST /api/reports/{id}/approve      # 通过 → 进入"已锁定 / 审核归档"
POST /api/reports/{id}/reject       # 驳回（需 rejection_reason）
POST /api/reports/{id}/withdraw     # 撤回（仅后台主管，或审核员撤回自己的决议）
```

## 5. Testing Decisions

### 什么算"好的测试"

我们遵循以下原则：

1. **只测外部行为，不测实现细节**
   - ✅ 测试"提交时必填字段缺失，返回 400 + 缺失字段列表"
   - ❌ 测试"validation 函数内部用了 if-else 嵌套 3 层"
2. **测试应该独立、可重复、无副作用**
   - 每个用例创建独立的 report / 账号，不依赖其他用例
   - 测试后清理数据，或使用数据库事务回滚
3. **测试覆盖业务规则，不只是代码覆盖率**
   - 100% 行覆盖 ≠ 100% 业务覆盖
   - 重点覆盖 v0.5 中明确定义的所有规则（6 个异常编码、2 条阻断规则、13 种状态流转、3 角色权限矩阵）
4. **失败用例必须能复现**
   - 错误日志包含 report_id、user_id、时间戳，能定位到具体场景

### 测试分层策略

| 测试类型    | 工具 / 框架                          | 覆盖目标            |
| ------- | --------------------------------- | --------------- |
| 单元测试    | Jest（后端）/ Vitest（前端）             | 业务规则函数、工具函数    |
| 集成测试    | Jest + Supertest（后端）/ Vue Test Utils（前端） | API 端点 + 数据库    |
| E2E 测试  | Playwright                        | 完整用户流程         |
| 手动测试    | -                                 | UI / UX、边界场景    |
| OCR 验证测试 | 5 张样本发票                          | OCR 识别准确率 ≥ 95% |

### 各模块测试重点

| 模块           | 关键测试点                                                                              | 反推自 v0.5     |
| ------------ | ----------------------------------------------------------------------------------- | ------------ |
| `validation` | 11 个主表必填字段缺失时阻断；明细表 4 个必填字段缺失时阻断；明细表为空时阻断；明细行全删除时阻断                              | 第 10 节       |
| `ai-label`   | 5 类规则共 6 个异常编码各自的触发条件；正常标签的 8 条不命中条件；多异常同时命中的处理                                | 第 11-12 节    |
| `ocr`        | OCR 成功 / 失败 / 部分识别 3 种状态；重传后 L2 编辑字段保留、OCR 字段被覆盖；OCR 字段与 L2 字段的来源标记               | 第 9 节 + 第 15.4 节 |
| `approval`   | 13 种状态流转的合法性；非法流转被拒绝；状态权限矩阵的 3 角色 × 8 状态校验                                       | 第 17 节 + 第 18 节 |
| `auth`       | L2 看不到其他 L2 的单子；后台全局可见 + 区域筛选；主管不受区域限制；主管可撤回任意决议，审核员只能撤回自己；L2 仅能重传自己的单子           | 第 21 节       |
| `audit-log`  | 所有字段修改、AI 打标、审批、撤回都进日志；append-only 不能删除；操作类型完整覆盖                                 | 第 14 节       |

## 7. Further Notes

### 7.1 Demo 范围与假设

为快速验证 MVP 核心流程，本项目**先做 demo 再补完整 PRD**。Demo 阶段聚焦以下范围：

**Demo 演示的流程（必须跑通）：**
1. L2 上传发票 → OCR 自动预填 → L2 修改 → 提交 → AI 打标（正常 / 异常）
2. L2 重传发票：覆盖 OCR 字段、L2 编辑保留
3. 后台审核员审批异常单 → 通过 / 驳回
4. 后台审核员撤回自己的决议 → 单子回到"待 L2 确认" → L2 重新提交
5. 后台主管撤回任意审核员的决议 → 单子回到"待 L2 确认" → L2 重新提交
6. L2 数据隔离演示（L2 A 完全看不到 L2 B 的单子）
7. 后台区域筛选演示
8. 异常规则触发演示（重复发票、关键字段被修改、OCR 失败）

**Demo 不演示的功能：**
- L1 经销商相关功能（Out of Scope）
- 导出、统计、报表（Out of Scope）
- 短信 / 邮件通知（Out of Scope）
- 主管强制二次复核（Out of Scope）
- 多级审批（Out of Scope）

**Demo 假设：**

| 维度      | Demo 假设                                     |
| ------- | ------------------------------------------ |
| OCR 服务  | 百度智能云 OCR（生产可用），demo 阶段可临时切换为 mock 服务便于调试  |
| SSO    | 复用联想现有 SSO，demo 阶段可使用 mock SSO（5 个预生成测试账号）   |
| 部署      | Docker Compose 本地部署（1 后端 + 1 MySQL + 1 MinIO）   |
| 数据      | 5 张样本发票 + 10 张预生成报备单 + 5 个测试账号                 |
| 时长      | 半天 demo + 1-2 周准备                              |

### 7.4 术语表（Glossary）

| 术语                | 含义                                            |
| ----------------- | --------------------------------------------- |
| L1               | 联想销售渠道中的"订货经销商"（发票购买方）                     |
| L2               | 联想销售渠道中的"提货服务商"（发票销售方，本系统主要用户）              |
| L3               | 联想销售渠道中的更深层级经销商（本期不涉及）                      |
| OCR              | Optical Character Recognition，光学字符识别，用于识别发票图片 |
| AI 标签 / AI 打标    | 系统基于规则自动给出"正常 / 异常"的判断                      |
| 阻断校验             | 提交前的必填字段检查，缺字段时阻断提交                         |
| 状态机              | 报备单的 8 个状态及其流转规则                              |
| 已锁定 / 审核归档       | 审批通过后的终态，数据不可修改（主管可撤回）                      |
| RBAC             | Role-Based Access Control，基于角色的权限控制          |
| 主表 / 明细表         | 报备单的两部分：主表记录发票级信息，明细表记录商品行信息                |
| SSO              | Single Sign-On，单点登录                              |
| OIDC             | OpenID Connect，基于 OAuth 2.0 的身份认证协议           |
| OCR 重传           | L2 上传错了发票后重新上传并触发 OCR 的能力                     |
| AI 打标覆盖          | 6 个异常编码只要命中 1 条，AI 标签即为"异常"                  |
