# 联想渠道发票报备 AI 辅助审核系统｜后台归档数据看板与导出改造方案 v1.0

> ## 文档说明
>
> 本文档用于定义后台端新增“归档数据看板与 Excel 导出”能力。
>
> 本功能面向真实业务 App 前端，不是 Demo 功能。前端实现中不得保留 Demo 字样、演示账号快捷登录、演示数据假设、演示流程文案或仅为演示服务的固定交互。
>
> 本功能只面向已经审核通过并归档的报备业务数据，用于销售流向查看、型号报备统计和 Excel 明细导出，不用于审批过程分析。

---

## 1. 改造目标

在现有后台端新增归档数据看板和 Excel 导出能力，支持后台用户查看已归档报备数据，并按 L2 经销商、L1 购买方、产品具体型号进行汇总与明细追踪。

核心目标：

1. 查看某个产品具体型号的归档报备数量。
2. 查看该型号由哪些 L2 经销商卖出。
3. 查看这些 L2 经销商分别卖给哪些 L1 购买方。
4. 支持按照业务筛选条件导出 Excel。
5. Excel 导出保留每一条商品明细行，便于业务线下继续分析。

---

## 2. 功能定位

新增后台模块建议命名为：

```text
归档数据看板
```

该模块定位为业务数据查询与导出工具，而不是审批分析工具。

本模块不展示、不统计、不导出以下审批辅助或过程类信息：

- AI 标签；
- 异常原因；
- 报备状态；
- OCR 字段；
- 操作日志；
- 审批过程字段。

后端可使用状态字段筛选已归档数据，但该状态字段不作为前端筛选项，也不作为 Excel 导出字段。

## 3. 适用用户与权限

本功能面向后台用户开放：

| 用户类型 | 是否可访问 | 说明 |
| --- | --- | --- |
| 后台审核员 | 是 | 可查看归档数据看板，可导出 Excel |
| 后台主管 | 是 | 可查看归档数据看板，可导出 Excel |
| L2 用户 | 否 | 不进入后台归档数据看板 |

权限要求：

1. 前端菜单只对后台用户展示。
2. 后端接口必须校验当前用户角色。
3. L2 用户直接请求后台看板或导出接口时必须拒绝。
4. 导出操作需要记录审计日志。

---

## 4. 数据范围

本模块仅查询和导出已经审核通过并归档的数据。

数据范围要求：

1. 只包含已审核通过、已归档的报备单。
2. 不包含待审批、已驳回、已撤回、待 L2 确认等数据。
3. 不包含 OCR 数据。
4. 不包含操作日志。
5. 不包含审批过程字段。
6. 不包含 AI 标签、异常原因、报备状态等审核辅助字段。

后端查询应默认强制限定为归档数据。前端不提供报备状态筛选。

## 5. 业务口径

### 5.1 经销商与购买方

本模块中的“经销商”指：

```text
L2 / 销售方 / 卖出方
```

本模块中的“购买方”指：

```text
L1 / 发票购买方
```

销售流向口径为：

```text
L2 经销商 → L1 购买方 → 产品具体型号 → 数量、金额、发票明细
```

### 5.2 报备数量

报备数量使用商品明细表中的 `purchase_quantity`。

统计时必须精确到产品具体型号：

```text
按 product_specific_model 分组
汇总 purchase_quantity
```

### 5.3 明细粒度

Excel 导出和明细表的最小粒度为：

```text
一行 = 一条商品明细行
```

即 `report_item` 级别。

示例：

如果一张发票包含 3 个产品型号，则导出时应生成 3 行数据：

```text
发票 A + L2 经销商 A + L1 购买方 B + 型号 1
发票 A + L2 经销商 A + L1 购买方 B + 型号 2
发票 A + L2 经销商 A + L1 购买方 B + 型号 3
```

这样可以保留每一个 L2 经销商、每一个 L1 购买方、每一个产品具体型号的原始业务明细。

## 6. 数据看板

数据看板可以展示汇总统计，但不改变底层明细粒度。

### 6.1 页面结构

建议新增后台一级菜单：

```text
归档数据看板
```

页面结构：

1. 筛选区；
2. 核心指标区；
3. 汇总视图区；
4. 明细数据区；
5. Excel 导出入口。

### 6.2 筛选条件

第一版支持以下筛选：

- L2 经销商；
- L1 购买方；
- 产品具体型号；
- 商品名称；
- 省；
- 地市；
- 城市；
- 发票号。

时间筛选暂不在第一版中固定具体口径，需要后续向业务确认。

待确认的时间口径包括：

- 报备提交时间；
- 归档时间；
- 发票开票时间。

### 6.3 核心指标

数据看板顶部展示以下指标：

| 指标 | 说明 |
| --- | --- |
| 归档报备单数量 | 符合筛选条件的归档报备单数量 |
| 商品明细行数 | 符合筛选条件的商品明细行数量 |
| 报备数量合计 | `purchase_quantity` 合计 |
| 金额合计 | 明细金额合计 |
| 涉及 L2 经销商数量 | 去重统计 |
| 涉及 L1 购买方数量 | 去重统计 |
| 涉及产品具体型号数量 | 去重统计 |

### 6.4 汇总视图

第一版支持以下汇总视图：

1. 型号汇总；
2. L2 经销商汇总；
3. L1 购买方汇总；
4. L2 经销商 + L1 购买方 + 产品具体型号汇总。

#### 6.4.1 型号汇总

## 7. Excel 导出

### 7.1 导出目标

Excel 导出用于业务离线分析，最终产物为一个 Excel 文件。

Excel 文件中同时包含：

1. 明细 Sheet；
2. 汇总 Sheet。

明细 Sheet 是核心交付内容，必须保留每一条商品明细行。

### 7.2 导出范围

导出范围与当前筛选条件一致。

导出数据要求：

1. 只导出归档报备数据。
2. 明细最多导出 50,000 行。
3. 超过 50,000 行时，后端应拒绝导出或要求用户缩小筛选范围。
4. 字段完整展示，不做脱敏。
5. 发票号、统一社会信用代码等字段完整导出。
6. 不导出 OCR 字段。
7. 不导出操作日志。
8. 不导出审批过程字段。
9. 不导出 AI 标签、异常原因、报备状态。

### 7.3 Sheet 设计

不需要导出说明 Sheet。

#### Sheet 1：报备明细

一行对应一条商品明细行。

字段范围：

- 报备主表业务字段；
- 商品明细字段；
- L2 经销商字段；
- L1 购买方字段；
- 发票业务字段；
- 区域字段；
- 其他已存在且属于归档业务数据的字段。

明确不包含：

- OCR 字段；
- 操作日志字段；
- 审批过程字段；
- AI 标签；
- 异常原因；
- 报备状态。

基础字段建议：

| 字段 | 说明 |
| --- | --- |
| 报备单号 | 报备单唯一标识 |
| 发票号 | 发票号码，完整展示 |
| L2 经销商名称 | 销售方 / 卖出方 |
| L2 经销商统一社会信用代码 | 完整展示 |
| L1 购买方名称 | 发票购买方 |
| L1 购买方统一社会信用代码 | 完整展示 |
| 商品名称 | 商品明细中的商品名称 |
| 产品具体型号 | 商品明细中的具体型号 |
| 购买数量 | `purchase_quantity` |
| 明细金额 | 商品明细行金额 |
| 明细税额 | 商品明细行税额 |
| 明细价税合计 | 商品明细行价税合计 |
| 发票总金额 | 发票级金额 |
| 发票总税额 | 发票级税额 |
| 发票价税合计 | 发票级价税合计 |
| 省 | 区域字段 |
| 地市 | 区域字段 |
| 城市 | 区域字段 |
| 报备提交时间 | 如字段存在则导出 |
| 归档时间 | 如字段存在则导出 |

如果数据库中还存在其他业务字段，例如 L1 编码、L2 编码、渠道编码、客户编号、产品编码等，应一并导出。

#### Sheet 2：型号汇总

按产品具体型号汇总。

字段：

- 产品具体型号；
- 商品名称；
- 数量合计；
- 金额合计；
- 涉及 L2 经销商数量；
- 涉及 L1 购买方数量；
- 涉及报备单数量；
- 涉及发票数量。

#### Sheet 3：L2 经销商汇总

按 L2 经销商汇总。

## 9. 后端接口建议

接口命名建议围绕归档报备数据：

```text
GET  /api/archived-reports/summary
GET  /api/archived-reports/model-summary
GET  /api/archived-reports/l2-summary
GET  /api/archived-reports/l1-summary
GET  /api/archived-reports/flow-summary
GET  /api/archived-reports/detail
POST /api/archived-reports/export
GET  /api/archived-reports/export/{taskId}
```

接口职责：

| 接口 | 职责 |
| --- | --- |
| `summary` | 返回核心指标 |
| `model-summary` | 返回型号汇总 |
| `l2-summary` | 返回 L2 经销商汇总 |
| `l1-summary` | 返回 L1 购买方汇总 |
| `flow-summary` | 返回 L2 + L1 + 型号销售流向汇总 |
| `detail` | 返回商品明细行级别数据 |
| `export` | 创建 Excel 导出任务 |
| `export/{taskId}` | 查询导出状态和下载地址 |

导出建议采用异步任务，避免大数据量导出导致接口超时。

后端必须强制执行：

1. 仅查询归档数据；
2. 后台角色权限校验；
3. 导出 50,000 行上限；
4. 不返回 OCR、日志、审批过程和审核辅助字段；
5. 导出操作审计记录。

## 10. 导出审计

- 导出结果：成功 / 失败；
- 导出数据行数；
- 导出文件名；
- 失败原因，如有。

## 13. 验收标准

### 13.1 数据范围

- 只展示已审核通过并归档的数据。
- 不展示待审批、已驳回、已撤回、待 L2 确认等数据。
- 不展示 AI 标签、异常原因、报备状态。
- 不展示 OCR 字段、操作日志和审批过程字段。

### 13.2 统计口径

- 型号报备数量等于商品明细表中同型号 `purchase_quantity` 合计。
- 一张发票多个型号时，统计正确拆分。
- 同一型号多行时，统计正确合并。
- L2 经销商指销售方 / 卖出方。
- L1 购买方指发票购买方。

### 13.3 Excel 导出

- 导出文件包含报备明细 Sheet。
- 导出文件包含型号汇总 Sheet。
- 导出文件包含 L2 经销商汇总 Sheet。
- 导出文件包含 L1 购买方汇总 Sheet。
- 导出文件包含 L2 + L1 + 型号汇总 Sheet。
- 不包含导出说明 Sheet。
- 报备明细 Sheet 一行对应一条商品明细行。
- 如果一张发票包含 3 个产品型号，导出中应出现 3 行明细。
- 明细最多导出 50,000 行。
- 超过 50,000 行时无法直接导出，并提示用户缩小筛选范围。
- 发票号、统一社会信用代码等字段完整展示。
- 不导出 OCR 字段、操作日志、审批过程字段、AI 标签、异常原因、报备状态。

### 13.4 权限与审计

- 后台审核员可访问归档数据看板。
- 后台主管可访问归档数据看板。
- L2 用户不可访问归档数据看板接口。
- 每次导出均记录审计日志。

### 13.5 前端

- 页面不出现 Demo、演示、测试账号等文案。
- 不依赖固定演示数据。
- 不保留演示账号快捷登录。
- 页面具备加载、空数据、错误状态。
- 导出具备处理中、成功、失败反馈。
