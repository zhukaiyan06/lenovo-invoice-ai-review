## 2. 建设目标与边界

### 2.1 建设目标

后台端需要支撑以下真实业务目标：

1. 后台审核员处理 L2 提交的报备单。
2. AI 帮助审核员发现 OCR 原值与 L2 提交值差异、重复发票、OCR 失败等风险。
3. 审核员人工决定通过或驳回报备单。
4. 审核员可撤回自己做出的审批决议。
5. 后台主管可查看全员审批历史，并撤回任意审核员的决议。
6. 所有审批、驳回、撤回、导出等关键动作均可追溯。
7. 后台用户可查看已归档报备数据看板。
8. 后台用户可导出已归档报备数据 Excel，用于线下业务分析。

### 2.2 第一版范围

第一版后台建设包括：

- 后台登录后的统一布局和导航；
- 审批队列；
- 我的处理历史；
- 审批详情；
- 主管全员审批历史；
- 审批通过；
- 审批驳回；
- 审批决议撤回；
- 操作日志查看；
- 归档数据看板；
- 归档数据 Excel 导出；
- 角色权限和状态权限校验；
- 导出审计。

### 2.3 暂不纳入第一版

第一版暂不建设：

- Demo 账号快捷登录；
- 固定演示数据依赖；
- 演示话术页面；
- L2 侧数据看板；
- 主管撤回后直接修改字段；
- 主管强制二次复核；
- 自定义 BI 报表；
- 定时导出；
- 邮件或短信通知；
- OCR 字段导出；
- 操作日志导出；
- 审批过程字段导出；
- 数据脱敏配置；
- 区域地图；
- 多维交叉透视表；
- 时间筛选口径固定。

说明：归档数据看板和导出是否需要时间筛选，以及按报备提交时间、归档时间还是发票开票时间筛选，需要后续向业务确认。

| 后台审核员 | 查看待审队列、处理报备单、查看自己的处理历史、撤回自己的审批决议、查看归档数据看板、导出归档数据 |
| 后台主管 | 具备审核员能力；额外可查看全员审批历史、撤回任意审核员的审批决议 |

L2 用户不进入后台端，不可访问后台审批页面、归档数据看板和归档数据导出接口。

如果后续需要 L2 查看自身数据，应单独设计 L2 侧数据看板，不能复用后台全量看板。

## 4. 状态口径

后台审批继续沿用 PRD 中的统一状态。

| 状态 | 含义 | 后台动作 |
| --- | --- | --- |
| `待 L2 确认` | L2 可编辑但尚未进入后台审批 | 后台无审批动作 |
| `待人工审批-正常` | 已提交，AI 未发现异常 | 通过、驳回 |
| `待人工审批-异常` | 已提交，AI 命中至少一条异常规则 | 通过、驳回 |
| `已驳回` | 后台已驳回，等待 L2 修改后重新提交 | 原审核员可撤回；主管可撤回 |
| `已锁定 / 审核归档` | 后台已通过，数据锁定并进入归档数据范围 | 原审核员可撤回；主管可撤回 |
| `已撤回` | L2 主动撤回后的终态 | 后台无审批动作 |

合法流转：

```text
待人工审批-正常 ──通过──→ 已锁定 / 审核归档
待人工审批-异常 ──通过──→ 已锁定 / 审核归档

待人工审批-正常 ──驳回──→ 已驳回
待人工审批-异常 ──驳回──→ 已驳回

已锁定 / 审核归档 ──撤回决议──→ 待 L2 确认
已驳回             ──撤回决议──→ 待 L2 确认
```

归档数据看板和导出只读取 `已锁定 / 审核归档` 数据。

## 6. 审批队列与处理历史

### 6.1 审批队列

审批队列展示待人工审批数据：

- `待人工审批-正常`；
- `待人工审批-异常`。

默认排序：

1. 异常单优先；
2. 同类数据按提交时间升序。

列表字段：

- 报备单号；
- 发票号；
- 提交方；
- 价税合计；
- 状态；
- AI 标签；
- 异常原因摘要；
- 提交时间；
- 操作入口。

筛选能力：

- 状态；
- AI 标签；
- 提交方；
- 发票号；
- 区域字段，如已有；
- 提交时间，如后续确认。

### 6.2 我的处理历史

我的处理历史展示当前登录审核员或主管已经处理过的数据。

列表字段：

- 报备单号；
- 发票号；
- 提交方；
- 价税合计；
- 当前状态；
- 审批结果；
- 审批时间；
- 是否可撤回；
- 操作入口。

审核员只能撤回自己的审批决议。主管可以在全员审批历史中撤回任意审核员的审批决议。

### 6.3 全员审批历史

全员审批历史仅主管可访问。

列表字段：

- 报备单号；
- 发票号；
- 提交方；
- 价税合计；
- 当前状态；
- 审批结果；
- 审批人；
- 审批时间；
- 是否可撤回；
- 操作入口。

## 7. 审批详情

审批详情用于辅助人工判断。

页面内容：

- 报备单基本信息；
- 发票原件预览；
- AI 标签和异常原因；
- OCR 原值与 L2 提交值 Diff；
- 商品明细；
- 最近操作日志；
- 完整操作日志入口；
- 通过、驳回、撤回决议等操作按钮。

Diff 展示规则：

- OCR 与 L2 值一致：普通样式；
- OCR 与 L2 值不同：高亮展示；
- OCR 未识别、L2 人工补录：显示人工补录标记。

日志规则：

- 首屏展示最近 5 条；
- 查看全部使用抽屉或弹窗；
- 日志至少记录操作类型、操作人、时间、变更前值和变更后值。

## 8. 审批操作与权限

### 8.1 前端按钮规则

| 当前状态 | 审核员 | 主管 |
| --- | --- | --- |
| `待人工审批-正常` / `待人工审批-异常` | 通过、驳回 | 通过、驳回 |
| `已锁定 / 审核归档`，当前用户是原审批人 | 撤回决议 | 撤回决议 |
| `已锁定 / 审核归档`，当前用户不是原审批人 | 无 | 撤回决议 |
| `已驳回`，当前用户是原审批人 | 撤回决议 | 撤回决议 |
| `已驳回`，当前用户不是原审批人 | 无 | 撤回决议 |
| `待 L2 确认` / `已撤回` | 无 | 无 |

### 8.2 后端强制校验

前端按钮只负责交互体验，后端必须再次校验：

1. 当前用户必须是后台审核员或后台主管。
2. 只有待人工审批状态允许通过或驳回。
3. 驳回原因不能为空。
4. 审核员只能撤回自己做出的决议。
5. 主管可以撤回任意审核员的决议。
6. 只有已驳回或已锁定归档状态允许撤回审批决议。
7. 重复通过、重复驳回、重复撤回必须拒绝。
8. 每次成功操作必须在同一后端事务中更新状态并追加日志。

建议接口：

```text
GET  /api/reports/review/queue
GET  /api/reports/review/history
GET  /api/reports/review/history/all
GET  /api/reports/{id}
POST /api/reports/{id}/approve
POST /api/reports/{id}/reject
POST /api/reports/{id}/review-withdraw
```

`review-withdraw` 用于区分后台撤回审批决议和 L2 主动撤回报备单。

## 9. 归档数据看板

### 9.1 数据范围

归档数据看板只读取已审核通过并归档的数据。

不展示：

- 待审批数据；
- 已驳回数据；
- 已撤回数据；
- 待 L2 确认数据；
- AI 标签；
- 异常原因；
- 报备状态；
- OCR 字段；
- 操作日志；
- 审批过程字段。

### 9.2 业务口径

经销商指：

```text
L2 / 销售方 / 卖出方
```

购买方指：

```text
L1 / 发票购买方
```

销售流向口径：

```text
L2 经销商 → L1 购买方 → 产品具体型号 → 数量、金额、发票明细
```

报备数量口径：

```text
按 product_specific_model 分组
汇总 report_item.purchase_quantity
```

### 9.3 筛选条件

第一版支持：

- L2 经销商；
- L1 购买方；
- 产品具体型号；
- 商品名称；
- 省；
- 地市；
- 城市；
- 发票号。

时间筛选口径待业务确认，第一版不固定。

### 9.4 核心指标

| 指标 | 说明 |
| --- | --- |
| 归档报备单数量 | 符合筛选条件的归档报备单数量 |
| 商品明细行数 | 符合筛选条件的商品明细行数量 |
| 报备数量合计 | `purchase_quantity` 合计 |
| 金额合计 | 明细金额合计 |
| 涉及 L2 经销商数量 | 去重统计 |
| 涉及 L1 购买方数量 | 去重统计 |
| 涉及产品具体型号数量 | 去重统计 |

### 9.5 汇总视图

第一版支持：

1. 型号汇总；
2. L2 经销商汇总；
3. L1 购买方汇总；
4. L2 经销商 + L1 购买方 + 产品具体型号汇总。

## 10. Excel 导出

### 10.1 导出范围

Excel 导出只导出归档业务数据。

导出范围与当前看板筛选条件一致。

导出要求：

1. 明细最多导出 50,000 行。
2. 超过 50,000 行时，后端拒绝导出或要求用户缩小筛选范围。
3. 字段完整展示，不做脱敏。
4. 发票号、统一社会信用代码等字段完整导出。
5. 不导出 OCR 字段。
6. 不导出操作日志。
7. 不导出审批过程字段。
8. 不导出 AI 标签、异常原因、报备状态。
9. 不需要导出说明 Sheet。

### 10.2 明细粒度

Excel 明细 Sheet 的粒度固定为：

```text
一行 = 一条商品明细行
```

如果一张发票包含 3 个产品型号，导出时应生成 3 行明细。

### 10.3 Sheet 设计

Excel 文件包含以下 Sheet：

| Sheet | 说明 |
| --- | --- |
| 报备明细 | 一行对应一条商品明细行，是核心交付内容 |
| 型号汇总 | 按产品具体型号汇总 |
| L2 经销商汇总 | 按 L2 经销商汇总 |
| L1 购买方汇总 | 按 L1 购买方汇总 |
| L2 + L1 + 型号汇总 | 按销售流向汇总 |

#### 10.3.1 报备明细 Sheet

字段范围：

- 报备主表业务字段；
- 商品明细字段；
- L2 经销商字段；
- L1 购买方字段；
- 发票业务字段；
- 区域字段；
- 其他已存在且属于归档业务数据的字段。

基础字段建议：

- 报备单号；
- 发票号；
- L2 经销商名称；
- L2 经销商统一社会信用代码；
- L1 购买方名称；
- L1 购买方统一社会信用代码；
- 商品名称；
- 产品具体型号；
- 购买数量 `purchase_quantity`；
- 明细金额；
- 明细税额；
- 明细价税合计；
- 发票总金额；
- 发票总税额；
- 发票价税合计；
- 省；
- 地市；
- 城市；
- 报备提交时间，如字段存在；
- 归档时间，如字段存在。

如果数据库中存在其他业务字段，例如 L1 编码、L2 编码、渠道编码、客户编号、产品编码等，应一并导出。

### 10.4 导出接口

建议接口：

```text
GET  /api/archived-reports/summary
GET  /api/archived-reports/model-summary
GET  /api/archived-reports/l2-summary
GET  /api/archived-reports/l1-summary
GET  /api/archived-reports/flow-summary
GET  /api/archived-reports/detail
POST /api/archived-reports/export
GET  /api/archived-reports/export/{taskId}
```

导出建议采用异步任务，避免大数据量导出导致接口超时。

## 14. 验收标准

### 14.1 前端正式化

- 页面不出现 Demo、演示、测试账号等文案。
- 不保留演示账号快捷登录。
- 不依赖固定演示数据。
- 页面具备加载、空数据、错误状态。

### 14.2 审批流程

- 审核员可以通过待审单。
- 审核员驳回时必须填写原因。
- 审核员可以撤回自己的通过决议。
- 审核员可以撤回自己的驳回决议。
- 审核员撤回其他审核员决议返回拒绝。
- 主管可以撤回任意审核员决议。
- 非法状态流转返回拒绝。
- 成功操作后状态、审批人和日志同时更新。

### 14.3 权限

- 审核员不可访问全员审批历史。
- 主管可访问全员审批历史。
- L2 不可访问后台审批接口。
- L2 不可访问归档数据看板和导出接口。

### 14.4 归档数据看板

- 只展示已审核通过并归档的数据。
- 不展示待审批、已驳回、已撤回、待 L2 确认等数据。
- 不展示 AI 标签、异常原因、报备状态。
- 型号报备数量等于商品明细表中同型号 `purchase_quantity` 合计。
- 一张发票多个型号时，统计正确拆分。
- 同一型号多行时，统计正确合并。

### 14.5 Excel 导出

- 导出文件包含报备明细 Sheet。
- 导出文件包含型号汇总 Sheet。
- 导出文件包含 L2 经销商汇总 Sheet。
- 导出文件包含 L1 购买方汇总 Sheet。
- 导出文件包含 L2 + L1 + 型号汇总 Sheet。
- 不包含导出说明 Sheet。
- 报备明细 Sheet 一行对应一条商品明细行。
- 如果一张发票包含 3 个产品型号，导出中应出现 3 行明细。
- 明细最多导出 50,000 行。
- 超过 50,000 行时无法直接导出，并提示用户缩小筛选范围。
- 发票号、统一社会信用代码等字段完整展示。
- 不导出 OCR 字段、操作日志、审批过程字段、AI 标签、异常原因、报备状态。
- 每次导出均记录审计日志。
